Test 2 - SQL Best Practices for Data Analysts

SQL Tutorials January 20, 2025 • 7 min read

Welcome to my second test blog post! In this article, I'll share essential SQL best practices that have helped me during my data analysis projects and my #100DaysOfAnalytics journey.

"SQL is the superpower of data analysts. Master it, and you unlock the ability to talk directly to your data."

1. Always Use Aliases for Tables

Using table aliases makes your queries more readable and easier to write:

-- Instead of this
SELECT employees.name, departments.department_name
FROM employees
JOIN departments ON employees.department_id = departments.id;

-- Do this
SELECT e.name, d.department_name
FROM employees e
JOIN departments d ON e.department_id = d.id;

2. Be Specific with SELECT Statements

Avoid using SELECT * in production queries. Specify only the columns you need:

-- Bad practice
SELECT * FROM customers;

-- Good practice
SELECT customer_id, first_name, last_name, email, signup_date
FROM customers;

Benefits of being specific:

  • Improves query performance
  • Reduces network traffic
  • Makes code more maintainable
  • Prevents breaking changes when schema evolves

3. Master JOIN Types

Understanding different JOIN types is crucial for any data analyst:

  • INNER JOIN: Returns matching records from both tables
  • LEFT JOIN: Returns all records from left table + matches from right
  • RIGHT JOIN: Returns all records from right table + matches from left
  • FULL OUTER JOIN: Returns all records when there's a match in either table

4. Use CTEs for Complex Queries

Common Table Expressions (CTEs) make complex queries more readable:

WITH monthly_sales AS (
    SELECT 
        DATE_TRUNC('month', order_date) AS month,
        SUM(amount) AS total_sales
    FROM orders
    GROUP BY 1
),
sales_growth AS (
    SELECT 
        month,
        total_sales,
        LAG(total_sales) OVER (ORDER BY month) AS prev_month_sales
    FROM monthly_sales
)
SELECT 
    month,
    total_sales,
    ROUND((total_sales - prev_month_sales) / prev_month_sales * 100, 2) AS growth_pct
FROM sales_growth;

5. Optimize WHERE Clauses

Filter early to reduce the dataset size before performing expensive operations:

-- Less efficient
SELECT * FROM (
    SELECT customer_id, SUM(amount) AS total
    FROM orders
    GROUP BY customer_id
) subq
WHERE total > 1000;

-- More efficient
SELECT customer_id, SUM(amount) AS total
FROM orders
WHERE amount > 0
GROUP BY customer_id
HAVING SUM(amount) > 1000;

6. Use Window Functions

Window functions are powerful tools for analytics:

-- Calculate running total
SELECT 
    order_date,
    amount,
    SUM(amount) OVER (ORDER BY order_date) AS running_total
FROM orders;

-- Rank customers by purchase amount
SELECT 
    customer_id,
    total_spent,
    RANK() OVER (ORDER BY total_spent DESC) AS spending_rank
FROM customer_totals;

7. Index Strategically

Create indexes on columns frequently used in:

  • WHERE clauses
  • JOIN conditions
  • ORDER BY statements

8. Document Your Queries

Use comments to explain complex logic:

/*
 * This query calculates monthly recurring revenue (MRR)
 * Excludes cancelled subscriptions and trial accounts
 * Used for: Executive Dashboard - Revenue Metrics
 */
SELECT 
    DATE_TRUNC('month', subscription_start) AS month,
    COUNT(*) AS new_subscriptions,
    SUM(monthly_price) AS new_mrr
FROM subscriptions
WHERE status != 'cancelled'
    AND plan_type != 'trial'
GROUP BY 1;

Conclusion

Following these SQL best practices will help you write cleaner, more efficient, and maintainable queries. Practice these techniques on real datasets, and they'll soon become second nature!
